Вот подробный сценарий использования (Use Case) приложения, разделенный на действия пользователя и соответствующие им технические процессы на бэкенде.

Юз-кейс: «Утилизация неопознанного мусора»
1. Регистрация и настройка локации
Действие пользователя: Открывает приложение впервые, вводит свой почтовый индекс (например, 160-0022).

Технический процесс:

Frontend (Flutter) отправляет запрос на Backend (FastAPI).

Бэкенд обращается к БД индексов (обработанный KEN_ALL_ROME.CSV).  

Находит соответствие: 160-0022 ➔ Tokyo, Shinjuku-ku.  

В PostgreSQL создается профиль пользователя с привязкой к district_id: shinjuku.

На смартфон возвращается подтверждение района.

2. Подготовка к анализу
Действие пользователя: Нажимает кнопку «Распознать мусор». Видит экран камеры с наложенной рамкой и подсказкой: «Положите предмет на ровную поверхность, убедитесь в наличии освещения».

Технический процесс:

Frontend активирует камеру. Включается алгоритм предобработки (фокусировка, проверка яркости), чтобы снизить количество неудачных запросов к дорогому Vision API.

3. Фотофиксация и отправка данных
Действие пользователя: Делает фото черного пластикового ведра и нажимает «Анализировать».

Технический процесс:

Фото сжимается и отправляется как multipart/form-data на эндпоинт /analyze.

Backend временно сохраняет изображение в S3 (Object Storage) и извлекает district_id пользователя из сессии.

4. Этап 1: Vision-анализ («Глаза»)
Процесс на стороне бэкенда:

FastAPI отправляет фото в OpenAI GPT-4o API (Запрос №1).

Промпт: «Identify object, material, estimated size, and cleanliness. Return JSON».

AI возвращает JSON: { "object": "bucket", "material": "plastic", "size_cm": 28, "clean": true }.  

5. Этап 2: RAG-поиск («Знания»)
Процесс на стороне бэкенда:

LangChain берет district_id: shinjuku и формирует запрос к Векторной БД (Qdrant).

Фильтрация: Поиск идет только по чанкам с метаданными district == "shinjuku".  

Результат: Извлекается текст: «С апреля 2024 года изделия из 100% пластика менее 30 см — это Ресурсный мусор. Сбор раз в неделю».  

6. Этап 3: Финальный вердикт («Мозг»)
Процесс на стороне бэкенда:

Формируется Запрос №2 к LLM (текстовый).

Промпт: Соединяет данные Vision-анализа (ведро, 28 см) и данные RAG (правила Синдзюку).  

ИИ выполняет логическую проверку: 28 см <= 30 см ➔ Категория: Ресурсы.  

7. Получение результата
Отображение в приложении:

Пользователь видит карточку: «Это Пластиковый ресурс. Положите в прозрачный пакет».  

Дополнительно: Приложение показывает дату следующего сбора для этой категории на основе календаря района (вытащенного из БД).

Резюме технического цикла:
Input: Image + Zip Code.  

Location Mapping: Zip Code ➔ District Name.  

Visual Perception: Image ➔ Structural JSON.  

Knowledge Retrieval: District Name ➔ Specific Garbage Rules.  

Reasoning: JSON + Rules ➔ Final Recommendation.  

UX: Финальный совет на языке пользователя.
