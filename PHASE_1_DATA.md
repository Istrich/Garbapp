# Phase 1: Data & RAG Infrastructure — Technical Specification

## 🎯 Общая цель
Создать фундамент системы: базу данных для сопоставления почтовых индексов с районами и векторное хранилище (RAG) с правилами утилизации мусора, учитывающее обновление правил от апреля 2024 года.

---

## 🛠 1. Модуль: Zip-Code Mapping (Обработка индексов)
**Агент:** `@DataArchitect`
**Файл данных:** `KEN_ALL_ROME.CSV`

### Задачи:
- Написать Python-скрипт `scripts/process_zip_codes.py`[cite: 1].
- Чтение CSV: Использовать кодировку `shift_jis` (обязательно для японских почтовых файлов)[cite: 1].
- **Логика трансформации:**
    - Извлечь колонку `0` (Почтовый индекс) и колонку `5` (Название муниципалитета латиницей)[cite: 1].
    - Нормализовать названия: убрать лишние пробелы, привести к нижнему регистру и `snake_case` (например, `SHINJUKU KU` -> `shinjuku`).
- **Хранение:**
    - Создать базу данных SQLite `data/zip_codes.db`.
    - Создать таблицу `zip_mapping` с полями `zip_code` (TEXT, PK) и `district_id` (TEXT).
    - Создать индекс по полю `zip_code` для обеспечения скорости поиска < 50мс.

### Критерии приемки (Acceptance Criteria):
- [ ] Скрипт успешно отрабатывает без ошибок кодировки.
- [ ] Запрос `SELECT district_id FROM zip_mapping WHERE zip_code = '1600022'` возвращает `shinjuku`.

## 📄 2. Модуль: RAG Knowledge Base (Подготовка контента)
**Агент:** `@DataArchitect`
**Файл данных:** `000378895.pdf` (Инструкция Синдзюку)[cite: 1]

### Задачи:
- Извлечь текст и таблицы из PDF с помощью `PyMuPDF4LLM` или аналогичных инструментов.
- Сформировать файл `knowledge/shinjuku_rules.md`.
- **Контентная разметка:**
    - Разбить на разделы: `Burnable`, `Non-burnable`, `Resources (Plastic)`, `Sodai Gomi`.
    - **Критическое требование:** Отразить правило апреля 2024 — изделия из 100% пластика размером до 30 см теперь утилизируются как **Resources (Пластик)** вместе с упаковкой[cite: 1].
    - Сохранить примеры предметов (ведра, вешалки, зубные щетки) в текстовом виде[cite: 1].

### Критерии приемки:
- [ ] Markdown файл содержит структурированные данные по всем категориям.
- [ ] Таблицы из PDF корректно преобразованы в текстовые списки или Markdown-таблицы.

## 🐳 3. Модуль: Vector Infrastructure (Инфраструктура RAG)
**Агент:** `@DevOps`

### Задачи:
- Создать `docker-compose.yml` для запуска контейнера **Qdrant**.
- Написать скрипт `scripts/init_vector_db.py` для инициализации коллекции `garbage_rules`.
- **Загрузка данных:**
    - Использовать модель эмбеддингов `text-embedding-3-small` от OpenAI.
    - Загрузить чанки из `knowledge/shinjuku_rules.md`.
    - Каждому вектору добавить метаданные: `{"district_id": "shinjuku", "last_updated": "2024-04"}`[cite: 1].

### Критерии приемки:
- [ ] Контейнер Qdrant запущен и доступен по API.
- [ ] Тестовый поиск по фразе "черное пластиковое ведро" с фильтром `district_id: shinjuku` возвращает блок правил про пластик до 30 см[cite: 1].
